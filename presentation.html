03
Root Cause

The problem isn't the skill. It's the world that moved.

  • You ship a new feature — the skill still describes the old flow
  • You rename a folder — the skill references the old path
  • Your API changes — the skill still assumes the old contract
  • Three months pass — no one remembers what the skill was for

Context decay is silent. You won't notice until the skill fails in front of a student or a client.

The 5 Primitives

Each primitive has a different update clock.

PrimitiveUpdate freqChanges when
CLAUDE.md Every sprint Project context, team conventions, or scope shifts
Skills On ship Any feature or workflow the skill touches changes
Memory Monthly Decisions made, preferences changed, stale facts accumulate
Sub Agents On API change Tools, APIs, or delegation patterns the agent relies on change
Hooks Rarely Wired once, stable unless the shell environment changes
05
Concept 01 — CLAUDE.md

If CLAUDE.md is stale, everything downstream is blind.

  • What belongs here: project goals, repo structure, conventions, active priorities, team context
  • What doesn't: code patterns, git history, architecture docs — Claude can read those directly
  • Length limit: under 200 lines — it's a prompt, not documentation
  • Update trigger: any sprint that changes goals, structure, or team conventions

"A 5-minute CLAUDE.md update saves 30 minutes of Claude going in the wrong direction."

06
Concept 02 — Skills

Tie skill reviews to ships, not to calendars.

What causes drift
  • Product changes without a corresponding skill update
  • No one owns the skill after it's created
  • Skill scope creep — one skill doing too many things
  • No signal that the skill is out of date until it fails
What prevents it
  • Add Last updated: header to every SKILL.md
  • Add skill review to your ship checklist
  • Keep each skill focused — one workflow, one command
  • Delete skills unused in 60 days
07
Concept 03 — Memory

Memory that's never pruned becomes a source of lies.

  • Auto memory captures decisions and preferences across sessions — great on day 1
  • Over time: conflicting entries, stale project context, dead-end notes accumulate
  • Claude can't tell the difference between a current preference and a 6-month-old one
  • Fix: 10-minute monthly prune — read every memory file, delete anything stale
  • Goal: accurate memory, not comprehensive memory
08
Concept 04 — Agents & Hooks

Hooks are set-and-forget. Agents follow their APIs.

Hooks (stable)
  • PreToolUse / PostToolUse / Stop / Notification
  • Wire once, test once, leave alone
  • Review only if shell environment changes
  • Most common failure: path or env var changes
Agents (API-coupled)
  • Delegate to specialist workers with isolated context
  • Review when upstream APIs or tools change
  • Review when the workflow they handle changes scope
  • Document which APIs each agent calls
09
The Update Trigger System

Four events that force a review before things break.

01
New feature shipped
Any skill that touches the changed workflow gets reviewed, updated, or deleted.
02
Workflow changed
If how your team works changes — new tools, new structure — audit CLAUDE.md and agents.
03
Skill failed
A skill that produced a bad output is a trailing signal. Fix it — then ask: what else has drifted?
04
Monthly audit
10 minutes. Review memory files, check skill headers, delete anything unused in 60 days.

How It All Fits — The Freshness Pyramid

Maintain the layers in order. Start at the top.

CLAUDE.md Project context, goals, conventions Every sprint
Skills Reusable workflows triggered by slash command On ship
Memory Decisions, preferences, user context Monthly
Sub Agents Specialist workers with isolated context On API change
Hooks PreToolUse / PostToolUse / Stop / Notification Rarely

Your Next 3 Steps

Today, this week, and this month.

1
Today — Audit your CLAUDE.md (5 min)
Is the project context still accurate? Are the active priorities current? Fix anything stale now. This has the highest leverage of anything in this deck.
2
This week — Add one update trigger to your ship checklist
Pick the trigger that fits your workflow: "when we ship, review the skills that touch the changed area." Add it as a checklist item. That's it.
3
This month — Delete one stale skill
Find a skill unused in 60 days, or one whose SKILL.md no longer matches reality. Delete it. Fewer, accurate skills beat many drifting ones every time.
